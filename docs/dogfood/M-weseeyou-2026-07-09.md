## Known v0.2 gaps (plugin backlog, not fixed this session)

- **Parameterized-path probing.** Verify has no strategy for hosting-rewrite wildcards (`*`) — needs either a synthesized placeholder ID with acceptance of domain-404s, or a `kind`/route flag marking "requires a real ID vibe-access can't supply cold." Needs input-schema modeling to do this properly.
- **Optional-auth modeling.** `detectAuth()` is a blind regex presence-check for `verifyAuthToken(`/`verifyIdToken(`; it can't distinguish a hard auth gate from a `try { verifyAuthToken() } catch { /* continue anonymously */ }` optional-auth pattern. Causes false `token` tags (and false verify fails) on intentionally-public routes with optional personalization.
- **Reset / capture / discovery-route scaffolds.** The six-need gap analysis (`gaps` command) still lists `agent-reset`, `agent-capture`, and `agent-manifest` (discovery route) as open; only `agent-seed` was scaffolded this session, scoped to the one gap picked for phase 2.

## Lessons

**(a) Structural-green != works.** The very first run — schema-valid config, schema-valid scan output, schema-valid manifest — was *completely* empty of real content (0 routes, 0 affordances) against a real app with ~85 endpoints. Five of the seven bugs were entirely invisible to the plugin's own fixture-backed test suite until pointed at a real codebase. Passing tests proved the plugin ran; it took the real app to prove the plugin worked.

**(b) The false-green pattern.** `e3d4609` is the sharpest case: it shipped with a fixture (`profiles.js`, bulk-export with plain function declarations), passed 68/68, and looked like the auth-detection fix. Against the real app it fixed nothing — WeSeeYou's actual handlers use the `const name = onRequest({...}, handler)` wrapper-call idiom, one level of nesting the fixture never exercised. A fix is only as honest as the fixture it's tested against; if the fixture doesn't match the real idiom, green tests certify nothing. The countermeasure that closed this out (`cd7b066`'s `moderation/{index.js,worker.js}` fixture) is explicit about it: "reproduces the exact real-world shape … skeleton copied verbatim from `pipeline.js`, sanitized." Verbatim-idiom fixtures, not abstracted stand-ins, are the guard against false-green.

**(c) Five scan-fix-rescan loops in one session.** Phase 1 (bugs 1+2, found and fixed) → phase 1c (bug 3, found and fixed, false-green) → phase 1f (bugs 4, 5, 6 surfaced and closed across the run) → phase 2 initial verify (bug 7 + the fantasy.js app bug surfaced) → phase 2b (bug 7 fix verified, fail count re-classified from 9 mixed causes down to 7 app-bug + 2 known plugin gaps). Every loop ran the same three-step discipline: scan/verify against the real app, root-cause by reading source (never guessed), fix with a failing-first test.
